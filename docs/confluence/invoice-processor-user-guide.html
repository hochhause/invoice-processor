1. What this tool does

The Invoice Processor turns supplier invoice PDFs into ready-to-upload bank payment files. You upload invoices, the tool reads them automatically, you double-check anything it was unsure about, and at the end you download one payment file (pain.001 XML) per bank, ready for e-banking.

It reads invoices in two ways:

  • Swiss QR code — if the invoice has a Swiss QR payment part, the data is taken directly from the code. This is exact and instant.
  • AI extraction — if there is no QR code (or fields are missing), an AI reads the invoice text and page images and fills in the payment details. AI results are good but not infallible — that is why the review step exists.

In short: Upload → AI Extraction → Review → Export.

3. How to use it, step by step

Step 1 — Upload invoices

  • Click + Upload PDFs or simply drag PDF files anywhere onto the page.
  • Only .pdf files are accepted — other file types are ignored without a message.
  • Each invoice appears as a row in the table. Invoices with a Swiss QR code are read immediately; the rest wait for AI extraction.

Step 2 — Run AI extraction

  • When there are invoices waiting, a ✦ Run AI Extraction button appears at the top. Click it once.
  • The table updates automatically every couple of seconds while the AI works. You do not need to refresh the page.
  • When it finishes, each invoice is either complete (LLM-Done) or flagged for your attention (Needs review).

Step 3 — Review and correct

  • Click the ✎ (pencil) icon on a row to open the invoice. You see the original PDF on one side and the editable payment fields on the other.
  • Field colours: red = required and still missing, green = filled, yellow = optional.
  • Four fields are always required before export: Receiver, IBAN, Amount, Currency.
  • The small chip next to the IBAN tells you where it came from:
    • doc ✓ — read from the invoice itself (QR or document); matches our records.
    • doc ⚠ — read from the invoice, but it differs from the IBAN we have stored for this supplier. Check carefully (see section 5).
    • DB — taken from our stored supplier list.
    • llm — found by the AI on the document.
    • manual — typed in by a person.
  • Use the coloured bank pills below the form to assign the invoice to a paying bank (e.g. BKB, Raiffeisen) or to MANUAL if it should not go into a payment file.
  • Click Save, or Save & Next to jump straight to the next invoice needing attention.

Step 4 — Export the payment files

  • Click Export → at the top. You get a board with one column per bank plus an "Unsorted" column.
  • Drag invoices into the bank column that should pay them. Invoices that are not ready (still in review, or no matching bank account configured) are locked and stay in MANUAL.
  • Click ✓ Accept & Download. The tool generates one pain.001 XML file per bank, bundles them into a ZIP, and downloads it. These XML files are what gets uploaded to e-banking.
  • Exported invoices are archived automatically — they disappear from the main view but are kept in the database.
  • If something still blocks the export, a popup lists exactly which invoices and what is missing — fix those on the dashboard first.

Other functions

  • Vendors — the stored supplier list (name + IBAN). The tool uses it to auto-fill and cross-check IBANs. You can add or update suppliers here.
  • Analytics — how many invoices were processed and what the AI usage cost.
  • CSV export — downloads the current invoice table as a spreadsheet.

Clear all permanently deletes every non-archived invoice for everyone, with no undo. During the test phase, please only use it when agreed with the team.

5. "Needs review" — why it happens and how to fix it

An invoice goes into Needs review whenever the tool is not confident enough to put it into a payment file on its own. This is a safety feature, not a malfunction. These are all the cases:

Case

Why it happens

How to fix it

AI could not read the invoice at all

Poor scan quality, unusual layout, or the AI service was unavailable.

Open the invoice (✎) and type the four required fields (Receiver, IBAN, Amount, Currency) by hand from the PDF shown next to the form.

A required field is missing

The AI read the invoice but could not find one of: Receiver, IBAN, Amount, Currency.

Fill in the fields shown in red, then save.

IBAN looks invalid

The IBAN failed the standard checksum — usually a misread digit.

Compare the IBAN character by character against the invoice and correct it. The form re-checks it when you save.

Foreign-currency invoice without BIC

Payments in currencies other than CHF need the receiving bank's BIC/SWIFT code, which was not on the invoice.

Find the supplier's BIC (it is often on the invoice footer or the supplier's website) and enter it in the BIC field.

IBAN differs from our supplier records (doc ⚠ chip)

The IBAN on this invoice is different from the IBAN we have stored for this supplier. This can be legitimate (supplier changed banks) — or a sign of a manipulated invoice.

Treat this seriously. Verify the new IBAN through a known channel (supplier portal, a phone number you already have — not one printed on the suspicious invoice). If the new IBAN is genuine, click ↑ Update vendor IBAN to update our records. If unsure, ask before paying.

Saving applies changes immediately — no restart needed. If a value is invalid, a red message under the save button explains which field to correct.

8. Security measures

This tool handles financial data (supplier IBANs, amounts, invoice documents). These are the protections in place during the testing phase, and what is planned for production:

Access control

  • The test link is restricted to the Lyfegen Microsoft 365 organisation. Every visitor must sign in with their personal corporate account — there is no shared password that could leak.
  • Access is granted and revoked centrally; it expires automatically after at most 30 days unless renewed. IT can disable the tunnel service organisation-wide by policy at any time.

Data in transit

  • All traffic between your browser and the tool is encrypted (HTTPS, TLS 1.2 or higher), relayed through Microsoft Azure infrastructure with Microsoft-issued certificates. Nothing travels unencrypted.

Data at rest

  • Uploaded invoices and all extracted data are stored only on the Lyfegen host machine (local database and file folder). The application itself stores nothing in any cloud.
  • Every invoice records an audit trail: where each IBAN came from (document, supplier database, AI, or manual entry), any mismatch against supplier records, and creation/modification timestamps.

AI processing

  • For extraction, the invoice's text and page images are sent to the Anthropic API (the company behind the Claude AI models) solely to read out the payment fields.
  • Under Anthropic's commercial terms, API data is not used to train AI models. Only usage metadata (token counts for cost tracking) is kept locally.
  • The API key is stored in a local settings file on the host machine and is never shown in the interface or sent anywhere except to Anthropic.

Fraud protection in the workflow

  • IBANs are checksum-validated; invalid ones block export.
  • If an invoice shows a different IBAN than our supplier records, it is flagged (doc ⚠) and held for human review — this is the primary defence against invoice-fraud attempts (see section 5).
  • Nothing is paid automatically: a human always reviews flagged invoices, assigns banks, and the final payment file still goes through the normal e-banking approval process.

Planned for production (Azure)

  • Deployment to a managed Azure environment behind Microsoft Entra ID sign-in and HTTPS.
  • Secrets (API keys) held in Azure Key Vault / App Settings instead of files.
  • The application additionally supports its own username/password layer (HTTP Basic auth), which will be enabled for any exposure beyond the organisation.

Known limitation of the test phase: all testers share one workspace — everyone sees all uploaded invoices, and the settings are shared. Production will revisit this. Please only upload invoices that all testers may see.
